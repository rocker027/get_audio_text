# CLAUDE.md

該文件在使用此存儲庫中的代碼時，為Claude Code（Claude.ai/code）提供了指導。

##存儲庫概述

這是一個基於BASH的音頻轉錄工具，可從各種平台（YouTube，Instagram，Tiktok等）下載音頻，並使用OpenAI的Whisper自動將其轉錄為文本。該工具被設計為用於音頻提取和轉錄的“一站式”自動化解決方案。

##核心體系結構

###主要組件

-  **`get_audio_text.sh` **：處理的主腳本：
- 通過YT-DLP下載音頻
- 通過耳語轉錄
- 文件管理和清理
- 用戶界面和錯誤處理

-  **`get-audio-text-extension/`**：空目錄（潛在擴展點）

###密鑰配置

- `audio_dir`：必須設置為絕對路徑（當前佔位符：`abs_path_to_audio_dir`）
- `transcript_dir`：自動設置為`
- 竊竊私語模型：`米（在腳本中可配置）

##通用命令

###設置命令
``bash
＃首次設置（用戶必須配置audio_dir路徑）
sed -i's | abs_path_to_audio_dir |/user/$ user/downloads/captureaudio | g'get_audio_text.sh
chmod +x get_audio_text.sh
````````

###依賴性安裝
``bash
釀造YT-DLP FFMPEG
PIP3安裝openai-whisper
````````

###用法命令
``bash
＃基本用法
./get_audio_text.sh“ https://www.youtube.com/watch?v=video_id”

＃跳過轉錄
./get_audio_text.sh“ url” -no-transcribe

＃轉錄後保留音頻文件
./get_audio_text.sh“ url”  -  keep-audio

＃完成後打開文件夾
./get_audio_text.sh“ url”  -  open-folder
````````

###測試/驗證
該腳本具有內置的依賴項檢查通過`check_dependencies（）`函數。不存在單獨的測試套件。

##開發筆記

###文件結構
- 下載到`$ audio_dir/``的音頻文件
- 將成績單保存到`$ audio_dir/tresscripts/`
- 臨時文件使用時間戳命名：`temp_audio_yyyymmdd_hhmmss`

###錯誤處理
- 對YT-DLP，FFMPEG和Whisper的強大依賴檢查
- 處理之前的文件存在驗證
- 自動清理臨時文件
-  YT-DLP下載的後備文件檢測

###支持平台
-  YouTube（主要）
-  Instagram（僅公共內容）
-Tiktok
- Facebook
-  YT-DLP支持的任何平台

###關鍵功能
- `check_dependencies（）`：驗證所需的工具
- `trentcribe_audio_with_rename（）`：處理竊竊私語和文件管理

##配置要求

在使用腳本之前，您必須：
1。更新`audio_dir`路徑從占位符到實際目錄
2.確保安裝所有依賴關係
3。使腳本使用`chmod +x`可執行

##耳語整合

該腳本支持多個耳語安裝路徑：
- 標準：`hisper'命令
-MacOS用戶安裝：`/user/rocker/library/python/3.9/bin/hisper`

輸出格式：TXT，SRT，VTT（所有三個自動生成）

# 開發偏好與工作流程

## 開發方法論
- **TDD 開發模式**：嚴格遵循 Red-Green-Refactor 循環
  - Red：先寫失敗的測試
  - Green：寫最少的程式碼讓測試通過
  - Refactor：重構程式碼，保持測試通過
- **漸進式開發**：偏好小步驟、小改動的開發方式
- **持續整合**：每個小改動都要確保系統穩定

## 需求釐清
- **主動確認**：如果需求不清楚或有歧義，必須先提出問題確認
- **理解優先**：優先確保理解正確，避免做錯方向的開發
- **邊界定義**：明確什麼要做、什麼不做

## 任務分解策略
- **自動拆解**：收到需求後，主動協助拆解為可執行的小任務（tasks）
- **任務規範**：每個 task 應該要：
  - 儘可能小且獨立（單一職責）
  - 有明確的編號（Task 1, Task 2, Task 3...）
  - 有清楚的目標和驗收條件
  - 可以在短時間內完成（建議 15-30 分鐘）
  - 包含具體的實作步驟
- **執行順序**：由使用者決定要先執行哪個 task
- **專注原則**：一次專注於一個 task，完成並驗證後再進行下一個
- **依賴管理**：如果 tasks 之間有依賴關係，要明確標示
